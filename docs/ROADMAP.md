# NovelAI Copilot Roadmap

## 路线原则

当前产品策略：

- 短期优先：继续完善前端本地闭环。
- 中期准备：以后端全量持久化为目标，但分阶段迁移。
- Prompt 方向：最终进入 Prompt Inspector，而不是只停留在提示词库。
- AI 工具方向：长篇记忆是最高战略优先级，但先做 V0 前置能力，不直接跳到完整 MemoryExtractionSkill。
- 写作能力方向：基础 AI 写小说能力必须保留，包括脑洞整理、大纲、章节续写、扩写、改写、摘要和卡文急救；但差异化不卷“自动生成整本书”，而是转向追读诊断、人味修订和可确认的章节改稿。

当前仍是前端 localStorage 闭环，不是后端数据库持久化，不是完整云同步，也没有真实模型调用。

## 2026-07-08 优先级重排与差异化定位

### 市场能力基线

通用 AI 写作 / 小说助手常见能力已经包括：

- 续写、扩写、改写、润色、摘要和纠错。
- 大纲、角色、世界观和章节规划。
- 长文本上下文、记忆或知识库。
- Web 搜索、Agent 操作和多步骤任务执行。
- 生图 / 封面 / 角色视觉参考。
- 多模态编辑、导出和工作流自动化。

NovelAI Copilot 不应只复制“AI 帮我写一段”的能力。产品差异化应集中在中文网文作者最在意的可持续连载与商业化判断：

- 爆款预测：题材、卖点、开篇钩子、主角目标、冲突密度、爽点节奏、章末追读的综合诊断。
- 追读诊断：不是预测真实平台追读率，而是给出可执行的章节修稿清单。
- 改写 / 扩写 / 精修：保留基础写作能力，但必须经过用户确认，不自动覆盖正文。
- 去模板腔 / 人味修订：定位为减少 AI 腔、说明腔、总结腔，增强人物反应、误判、欲望和网文口感；不承诺绕过检测。
- 长篇记忆：让 OOC、伏笔、人物状态和世界规则可确认、可追溯、可修正。
- IP 衍生：短剧脚本、互动剧情游戏设定包、漫剧 / 分镜 / 角色视觉等从正文衍生，而不是与写作工作台割裂。

### 短剧和游戏剧情是否排入路线

已排入，归入 `IP Factory`：

- P1.5：短剧脚本草案、互动剧情游戏设定包，优先围绕当前章节和作品设定生成。
- P2：漫剧分镜、角色视觉、场景图、封面、生图资产管理。
- P2 暂不做完整视频合成、配音、可运行游戏工程导出。

### 智能助手能力拆分

智能助手不只做聊天，后续拆成三层：

1. 搜索层：搜索作品、章节、设定、记忆、检查结果、产物记录。
2. 操作层：打开作品、定位章节、插入建议、生成摘要、提取记忆、运行检查、保存版本。
3. 代理层：根据用户目标自动编排多步任务，但每次写入正文、记忆或后端前必须用户确认。

### 生图能力排期

生图能力排入 P2，但先从写作强相关资产开始：

- 封面概念图。
- 角色卡 / 人设视觉参考。
- 场景氛围图。
- 短剧 / 漫剧分镜参考图。
- IP Factory 资产包封面和角色图。

不优先做泛娱乐图片社区，不把生图放在章节写作闭环之前。

## P0：本地闭环稳定期

目标：让用户可以在当前浏览器中稳定管理作品、章节、设定摘要和作品总览。

已完成：

- 作品库 / 项目管理页 P0。
- WorkManagementModal 作品管理弹窗。
- 作品总览 / 作品驾驶舱 P0。
- 本地作品库持久化：`localStorage:yixie-works-library-v1`。
- 作品库视图模式持久化：`localStorage:yixie-works-library-view-v1`。
- 故事线图本地数据：`localStorage:yixie-storyline-data-v1-${workId}`。
- 本地 E2E 验收清单和基础文档。

P0 仍需收口：

- Prompt 弹窗支持本次编辑、导入、恢复默认和复制当前 Prompt。
- 章节统计联动 P0.5：章节保存、删除、发布标记后，作品库和作品总览的字数、章节数、更新时间尽量保持一致。
- 本地数据异常兜底：继续清理 NaN、undefined、null 和缺失字段导致的页面错误。
- UI 文案继续去除误导性 AI 感：不把原型写成已接入，不把本地保存写成云端同步。

P0 不做：

- 后端持久化。
- 真实云同步。
- 真实模型 API。
- 真实平台发布。
- 复制、导出、归档、删除的完整真实逻辑。
- MemoryExtractionSkill。
- ConsistencyCheckSkill。
- Prompt Inspector。
- 追读率真实预测。
- 绕过 AI 检测式“反 AI”承诺。

## P1：可长期使用的轻量版本

目标：从“本地原型”进入“可以长期写作”的轻量产品。

### P1-A：作品管理轻 UX

- 保存成功提示。
- 关闭弹窗前的未保存提醒。
- 标签增删体验优化。
- 发布平台自定义输入。
- 管理弹窗字段分组和校验文案优化。
- 作品总览轻量编辑创作内容字段，例如一句话卖点、大纲摘要、世界规则摘要。

注意：作品管理弹窗负责项目管理字段；作品总览的轻编辑只处理创作资料字段，避免两个入口职责混乱。

### P1-B：安全数据动作

危险动作按顺序单独拆任务：

1. 导出 Markdown / JSON。
2. 复制作品。
3. 归档作品。
4. 删除作品。

这些动作不能混在 UI 优化任务里。删除必须最后做，并需要二次确认。

### P1-C：后端持久化 MVP

第一版后端目标：

- 登录用户可以创建作品。
- 保存作品基础信息。
- 保存章节。
- 重新登录后可以读取作品和章节。
- 游客草稿继续保留本地保存。
- localStorage 迁移到后端需要明确确认，不自动吞掉本地数据。

第一版暂不做：

- 多端冲突合并。
- 实时协作。
- 复杂版本历史。
- 平台发布授权。
- 完整云同步策略。

### P1-D：Prompt 工作流

先做 Prompt 轻量工作流：

- 展示本次上下文来源。
- 展示 Prompt 预览。
- 允许用户补充要求。
- 允许本次编辑 Prompt。
- 复制 Prompt。
- 记录最近一次 Prompt。
- AI 结果回填后必须由用户确认，不自动覆盖正文。

正式提示词库和 Prompt Inspector 放到后续阶段。

### P1-E：追读诊断 V0

追读诊断是新增的网文向核心工具。第一版不接真实平台数据，不承诺预测真实追读率，只做基于章节文本和网文经验规则的修稿诊断。

能力范围：

- 开头钩子诊断：前 300 字是否有问题、欲望、冲突或异常。
- 主角目标诊断：本章主角想要什么、阻碍是什么、读者是否能立刻理解。
- 冲突压力诊断：是否有明确对抗、期限、损失或情绪压力。
- 爽点兑现诊断：本章是否有阶段性获得、反击、打脸、揭示或关系推进。
- 信息密度诊断：是否有过多解释、铺垫太长、水段或重复表达。
- 章末悬念诊断：最后 300 字是否制造下一章阅读理由。
- 修改建议：给出问题位置、影响原因、修改方向和可替换示例。

V0 不做：

- 不承诺真实平台追读率预测。
- 不接平台后台数据。
- 不自动覆盖正文。
- 不把建议包装成唯一正确答案。

### P1-F：人味修订 V0

人味修订不叫“反 AI 检测”，定位为减少模板腔、增强作者表达和网文口感。

能力范围：

- 识别模板化句式、总结腔、说明腔、价值升华腔。
- 识别人物台词不像人、情绪太平、动作细节不足。
- 识别剧情过顺、缺少误判、犹豫、欲望和人性复杂度。
- 输出保守改、强刺激改、爽文改三种修订方向。
- 支持复制、插入、替换选中文本或保存为修稿记录。

边界：

- 不承诺规避 AI 检测。
- 不做批量洗稿。
- 所有修改必须用户确认。

## P1.5：长篇记忆 V0 前置能力

长篇记忆是战略优先级最高的官方工具，但不建议第一步直接实现完整 MemoryExtractionSkill。

P1.5 先做 Memory V0：

- 章节摘要。
- 事件摘要。
- 人物状态摘要。
- 伏笔 / 设定摘录。
- 来源章节记录。
- 置信度或用户确认状态。
- 用户确认后写入本地记忆。

追读诊断和人味修订可以读取 Memory V0 中的已确认人物状态、伏笔和世界规则，但不能自动写入或覆盖记忆。

Memory V0 的目标不是“AI 已经记住全书”，而是建立可确认、可追溯、可修正的记忆条目。

## P2：智能化和云端化阶段

P2 才进入完整 AI 副驾能力：

- MemoryExtractionSkill：从章节中抽取人物、事件、设定、伏笔和世界规则。
- ConsistencyCheckSkill：基于记忆、人物设定、章节内容进行 OOC 和一致性检查。
- Prompt Inspector：展示 Prompt 上下文来源、引用内容、变量、token 结构和风险。
- AI Copilot Chat：围绕当前作品上下文进行多轮协作。
- 故事线图与章节联动：从章节自动生成事件线，用户确认后写入。
- 后端同步详情：待同步、冲突、失败原因和恢复操作。
- 发布准备增强：简介、标题、标签、章节卖点和发布检查。
- 追读诊断增强：结合长篇记忆、章节目标、故事线、读者反馈和历史修稿记录，输出更可靠的章节级修稿建议。
- 人味修订增强：学习作者确认过的改稿偏好，形成作品级风格约束。

## 关键提醒

- 先作品闭环，再 AI 智能。
- 先章节和资料稳定，再长篇记忆。
- 先基础 AI 写作可用，再做追读诊断和人味修订。
- 先用户确认，再写入记忆。
- AI 结果不要自动覆盖正文。
- 状态文案必须诚实。
