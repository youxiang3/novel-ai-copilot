# Codex 任务规则

## 基本原则

- 每次只做一个小任务。
- 先读当前代码和文档，再判断改动范围。
- 优先沿用现有数据结构、组件和交互模式。
- 不要大范围重构。
- 不要格式化无关文件。
- 不要删除已有功能。
- 不要新增依赖，除非任务明确要求且说明原因。

## 禁止破坏的内容

小任务中不得破坏：

- WorkManagementModal。
- 作品总览 / 作品驾驶舱。
- 章节管理。
- 新建作品确认流程。
- 短画面扩写入口。
- 卡文急救入口。
- 世界观图谱入口。
- 故事线图入口。
- 作品库卡片视图 / 列表视图。
- 作品库右侧当前作品详情。
- 官方工具状态区。

## localStorage 规则

不得在普通任务中重命名、清空或迁移以下 key：

- `yixie-works-library-v1`
- `yixie-works-library-view-v1`
- `yixie-storyline-data-v1-${workId}`
- `yixie-phase3-workspace-${workId}`

如果未来确实需要迁移，必须单独开迁移任务，写清兼容策略、回滚策略和验收步骤。

## 当前产品边界

除非任务明确要求，否则不要实现：

- 后端持久化。
- 真实云同步。
- 真实模型 API。
- 真实平台发布。
- 复制作品。
- 导出作品。
- 归档作品。
- 删除作品。
- MemoryExtractionSkill。
- ConsistencyCheckSkill。
- Prompt Inspector。
- 大范围 StoryGraphCenter 扩展。

## 文案和状态规则

- 不要把本地保存说成云端已同步。
- 不要把原型工具说成真实模型能力。
- 游客草稿必须明确“仅本地保存”。
- 字段缺失时必须有空状态或默认值。
- 页面不应展示 NaN、undefined 或 null。

## 验证规则

涉及业务代码时，完成后需要运行：

```bash
cd frontend
npm run build
```

只改文档时可以不运行 build，但最终回复必须说明原因。

涉及交互时，需要说明手动验证路径，例如：

- 新建作品 -> 作品库 -> 管理弹窗 -> 保存 -> 作品总览 -> 刷新。
- 作品库卡片视图 / 列表视图切换。
- 总览进入章节管理。
- 故事线图入口。
- 工具入口。

## 状态文档规则

任务完成后需要更新 `docs/PROJECT_STATUS.md`，至少说明：

- 完成了什么。
- 修改了哪些文件。
- 是否改了业务代码。
- 是否运行 build。
- 当前仍未完成什么。
- 风险点。

## 最终回复规则

完成后必须输出：

- 修改文件。
- 每个文件改了什么。
- 是否影响已有流程。
- 是否运行 `npm run build` 及结果。
- 当前未完成项。
- 下一步建议。
